# v0.12.0 — Admin Control Center (2026-06-28)

An internal, **admin-only `/admin` portal** for running the product — gated behind
a new `users.is_admin` flag and invisible to everyone else (non-admins get a 404 /
redirect, never a hint it exists). Three tabs: **Overview** (publish calculators
live), **Users** (tiers + suspension), **Analytics** (usage, GA4-ready).

The headline architectural change: **publish state moved out of the frontend
bundle and into the database.** Calculators can now be published / unpublished
**live from the portal** with no redeploy — the public app reads the publish set
at runtime.

This cycle also folds in two independent improvements that landed alongside it:
**migrate-on-boot** (the schema now auto-applies on every deploy) and a small
sidebar brand-link fix.

## Admin auth (#149)
- `users.is_admin` (additive, idempotent; defaults false) is the single source of
  truth for who's an admin; admins are promoted manually (`User.set_admin`) — no
  self-serve. `/api/auth/status` surfaces the flag so the SPA can gate `/admin`.
- An `admin_required` decorator gates every `/api/admin/*` route: **401** with no
  session, **404** for a logged-in non-admin (the portal doesn't acknowledge its
  own existence). Status is read live from the DB each request, so a demotion
  takes effect immediately.

## Overview — live publish toggles (#150)
- New `calculator_publish` table (seeded from `calc_types.DEFAULT_PUBLISHED_TYPES`)
  is the **runtime** source of truth for the public surface. `GET /api/calculators/published`
  (public) is what the app reads; `PATCH /api/admin/calculators/:type` flips it.
- The frontend's published surface (sidebar, landing grid, marketing showcase,
  calculator route-guard) now derives from a runtime hook (`usePublished.js` — a
  module-store, no Context) with the registry's build-time defaults as a fallback
  if the fetch fails. Hard rule #3 was updated to match.
- The Overview screen: a calculator catalog with a stat strip and per-row toggles
  that publish/unpublish on the public `/app` immediately (optimistic + rollback).

## Users — tiers, suspension, audit (#151)
- New columns: `users.tier` (`free`/`pro`/`elite`, single-sourced in
  `user_tiers.py`), `users.suspended`, `users.last_login_at`; plus an append-only
  `admin_audit_log` (JSONB detail).
- `GET /api/admin/users` (search + tier filter + counts) and
  `PATCH /api/admin/users/:id { tier?, suspended? }` — audit-logged, CSRF +
  rate-limited, with a self-suspend guard. Suspended accounts are blocked at login
  (403, after the password check so it doesn't leak which emails exist).
- The Users screen: searchable, tier-filterable accounts table with an inline
  tier dropdown and suspend/reinstate (suspended rows dimmed + badged). During
  beta everyone is Free; tiers are manual comp with **no entitlement enforcement
  yet** (that's the deferred Freemium decision).

## Analytics — GA4 proxy with a DB-backed empty state (#152)
- `GET /api/admin/analytics` always returns real **signups + the Free/Pro/Elite
  funnel from our own database**; visitors / traffic sources / per-calculator runs
  come from **GA4 via a server-side proxy** (`services/analytics.py`) when
  configured — the service-account key stays server-side, never shipped to the
  client.
- Until GA4 is wired (`GA4_PROPERTY_ID` + `GA4_CREDENTIALS_JSON`), the tab shows a
  clean "connect GA4" empty state while the signup KPIs still render. The GA4 SDK
  is optional and lazily imported (commented in `requirements.txt`) so the default
  install stays lean.

## Also in this release
- **Migrate-on-boot (#148):** `backend/gunicorn.conf.py`'s `on_starting` hook runs
  the idempotent `db_init` once in the master before workers fork, so every deploy
  self-migrates. A failed migration aborts startup instead of serving a stale
  schema. `init_db` is advisory-locked so concurrent boots don't race. This closes
  the old "code shipped before the migration was run" gap — and means this
  release's new schema applies automatically.
- **Sidebar brand link (#147):** the `SpreadsheetMillionaire` wordmark in the app
  sidebar now returns to the homepage (`/`).

## Notes
- All schema changes are **additive and idempotent**.
- Going **live** with GA4 analytics later needs: the two env vars, uncommenting
  `google-analytics-data`, and a `calc_run` public event — the last is a
  privacy/consent decision and was deliberately left out of this release.
- The portal is internal; the public product (the four live calculators, the
  trackers' dark status) is unchanged.
