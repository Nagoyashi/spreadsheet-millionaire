# v0.15.0 — I&E bulk data entry: the monthly category grid (2026-07-14)

The cycle that makes the product live up to its name. Adding transactions
one-by-one is fine for a single purchase but tedious for a whole month of
bookkeeping — the Income & Expense tracker now has a **Monthly entry** tab: pick
a month, fill in each category's sum exactly like one month-column of a
bookkeeping spreadsheet, save once. One design decision ties the cycle together:
a grid cell is **one aggregate transaction row** (`source='monthly'`, first of
month) in the same `ie_transactions` stream — so the Overview, category
breakdowns, avg/median, and the recurrence forecast needed **no second
aggregation path**, and the audit that closed the cycle confirmed it. This
cycle also resequenced the roadmap: bulk entry took the v0.15 slot and billing
moved to v0.16.

## Data model + bulk month-entry API (#292)
- `ie_transactions.source` (`'manual' | 'monthly'`) — single-sourced in
  `income_expense_types.py`, added via additive idempotent DDL + CHECK rebuild.
  **Server-set per write path, never request input** (both schemas raise on an
  unknown `source` field). `'import'` joins the enum when CSV/PDF import ships.
- `GET /api/income-expense/months/<year>/<month>` — the month's aggregate cells
  plus read-only per-(type, category) sums of individually-entered rows,
  bucketed by month range so a date-edited row stays visible in its month.
- `PUT /api/income-expense/months/<year>/<month>` — replaces the month's
  aggregate rows **wholesale** (delete-then-insert in one transaction; an
  omitted cell is a cleared cell). No upsert bookkeeping, trivially idempotent;
  a partial unique index backstops one monthly cell per user+type+category+date.
- Validation: per-type categories, positive amounts, duplicate cells rejected,
  payload capped at the category-combo count, calendar-range check on the URL.
- Decision recorded in `DECISIONS.md` § "Income & Expense Tracker" (Monthly
  grid bulk entry) **before** implementation.

## Monthly entry grid UI (#293)
- New **Monthly entry** tab between Overview and Transactions: month/year
  picker (defaults to the current month; year options cover existing data plus
  a window around today for back-filling), Income + Expense sections side by
  side, one bounded `NumInput` per curated category.
- Spreadsheet keyboard flow: **Enter/Tab advances to the next field**; running
  section totals + net update as you type.
- No double-entry: existing cells prefill from the GET, and per-category sums
  of individually-entered transactions render read-only beneath the inputs.
- One **Save month** action; the page summary refetches through the hook's new
  `saveMonth`, which also fires the one-shot `trackerFirstEntry` activation
  event on a non-empty save. `MONTH_NAMES` single-sourced in
  `incomeExpenseOptions.js`.

## Overview + selectors treat grid rows correctly (#294)
- The audit confirmed the design goal: because grid cells are ordinary
  transactions, `categoryBreakdown`, `monthlyIncomeStats`, the recurrence
  forecast, and the backend year summary all handle them with **zero changes**.
- Aggregate rows are now recognizable where users meet them individually: an
  amber **"Monthly entry"** badge in the Transactions list (grid↔list edits
  stay consistent).
- The audited behavior is pinned by regression tests: selectors never filter on
  `source`; the forecast never projects an aggregate row and treats a
  bulk-entered month as real activity; the year summary folds grid rows into
  totals/by-month/by-category.

## Roadmap resequenced — billing moves to v0.16 (#295)
- `docs/ROADMAP.md`: Phase 15 is the bulk-entry cycle; **billing slides to
  Phase 16** and absorbs the free/premium boundary decision from the former
  "trackers go live" phase, which was overtaken by events — both trackers went
  live free during the v0.14.x rollout, so per invariant 9 today's live surface
  is the free tier's floor and premium must be additive.
- CSV import stays v0.16.1 (pre-filed #206–#210); **PDF bank-statement import
  (Kontoauszug)** filed as its unscheduled backlog complement (#296).
