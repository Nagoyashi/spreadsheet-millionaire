# v0.14.0 — Observability + analytics foundation (2026-07-11)

The first of three Phase-14 sub-releases (go-live readiness + instrumentation).
No new product surface — the goal is to make the *existing* app observable and
measurable before a data-driven free/premium boundary is set in a later phase.
Trackers stay dark. Everything here is **credential-gated**: with no keys set, dev,
CI, and a fresh checkout run with monitoring and analytics off, and nothing phones
home — so none of it is load-bearing for availability.

## Error monitoring (Sentry)
- **Backend (#173)** — unhandled Flask errors report to Sentry, DSN-gated on
  `SENTRY_DSN`. Initialised before the app is built so `FlaskIntegration` wraps
  request handling from the first request; `send_default_pii=False` keeps request
  bodies, cookies, and user ids off events. A missing DSN in production is a
  startup warning, never an exit.
- **Frontend (#174)** — `@sentry/react` reports browser crashes, DSN-gated on
  `VITE_SENTRY_DSN`. Wired into the two places an error is already caught: the
  render error boundary and the central 401-logout path. No Session Replay, no
  performance tracing, no PII.

## Structured request logging (#175)
- One structured line per request (method, path, status, duration) via Flask
  before/after hooks — JSON to stdout in production, a readable line in dev.
  Stdlib only, no new dependency. Each request gets an id echoed on
  `X-Request-ID` and tagged onto the Sentry scope, so a log line, a support
  report, and a Sentry event for the same request correlate. The access log
  carries no client IP, user id, body, or query string (privacy by construction).

## Uptime / health monitoring (#176)
- New **readiness probe** `GET /api/health/ready` verifies the one hard dependency
  (Postgres, via `SELECT 1`) → `200 {db:ok}` or `503 {db:down}`. It's the endpoint
  an external uptime monitor polls and alerts on. `GET /api/health` stays a
  deliberately dumb **liveness** probe so a transient DB blip never trips Render's
  restart logic. A failing readiness probe logs the DB error; successful polls of
  either are skipped to keep the log signal clean.

## Product analytics — the activation funnel (PostHog, EU cloud)
- **SDK + funnel events (#177)** — a first-party, EU-hosted PostHog SDK
  instruments the activation funnel `calculator_used → account_created →
  tracker_first_entry → second_session → upgrade_viewed → upgrade_clicked`.
  Key-gated on `VITE_POSTHOG_KEY`. Locked down for a finance app: autocapture off,
  Session Replay off, no pageview stream, `identified_only` profiles, and payloads
  carry only non-PII dimensions (never an email or a financial value). A minimal
  Settings "Premium" card gives the two upgrade events a real surface until
  billing exists. PostHog is added to the privacy page's sub-processor list.
- **Admin surface (#178)** — the `/admin` Analytics screen reads the funnel back
  **server-side** (HogQL Query API via stdlib `urllib`, a secret personal API key
  that never reaches the client). New "Activation funnel" card, real per-calculator
  usage from `calculator_used`, and vendor-aware status. Gated independently from
  the existing GA4 traffic proxy; a broken PostHog is labelled, not a 500.

## Operator setup (post-merge, optional)
Each integration activates only when its keys are set (all documented in
`docs/DEPLOYMENT.md`):
- **Sentry** — `SENTRY_DSN` (backend) + `VITE_SENTRY_DSN` (frontend build).
- **PostHog** — `VITE_POSTHOG_KEY` (browser ingest) + `POSTHOG_API_KEY` /
  `POSTHOG_PROJECT_ID` (backend read-back for the admin funnel). Use EU-region
  keys to keep event data in the EU.
- **Uptime** — point an external monitor at `/api/health/ready` with alerting.
