# v0.11.0 — Income & Expense Tracker (2026-06-21)

The second tracker. Log income and expenses, categorise spending, and see your
monthly and yearly cashflow at a glance. Like the Net Worth tracker it's
**production-ready but ships dark** — hidden behind a feature flag so production
keeps showing "Coming soon" until a later waiting-list launch.

This cycle also settled the architecture question the Net Worth cycle left open:
with two trackers now real, **no shared "tracker framework" is extracted** — they
reuse UI primitives (the config-driven form pattern, recharts, `httpClient`,
`fmt`, `NumInput`) and keep `trackers.js` as the only shared abstraction
(the published-tracker registry). Per-tracker page/api/hook stay separate because
the data models differ materially.

Modelled on the earlier `fintrackr_dev` cashflow module, adapted to this repo's
hard rules.

## Data model (#120)
- One normalised `ie_transactions` table — income/expense rows where the sign is
  carried by `type` (`amount > 0`), a curated `category`, and `occurred_on`.
  User-scoped (FK + `user_id` and `(user_id, occurred_on)` indexes), type and
  category enforced by named `CHECK` constraints.
- `backend/income_expense_types.py` is the single source of truth for the enums
  (transaction types + curated income/expense categories), imported by both the
  schema and the DDL — the `calc_types` / `net_worth_types` pattern.

## API (#121)
- `/api/income-expense/*` — transactions CRUD (list filterable by `?year=&month=`)
  and a SQL-aggregated `GET /summary`: a 12-month income-vs-expense series,
  totals, spend-by-category, and the list of years with data. Every query filters
  `user_id` (Hard Rule #6); writes are CSRF-protected and rate-limited; the schema
  validates that a category matches its transaction type.

## Frontend (#122, #123, #124)
- `incomeExpenseApi` + a `useIncomeExpenseData` hook (year/month-filtered
  transactions + summary; CRUD that refetches).
- A tabbed page at `/app/income-expenses` with a sticky Income / Expense / Net bar.
- **Transactions** panel: year/month/type filters, a signed-amount table, and an
  add/edit form whose category options follow the selected type.
- **Overview** dashboard on recharts: summary cards with savings rate, a monthly
  income-vs-expense bar, and a spending-by-category pie.

## Ships dark (#122 / #125)
- A second feature flag, `INCOME_EXPENSE_ENABLED` (env `VITE_INCOME_EXPENSE_ENABLED`,
  default `import.meta.env.DEV`), adds the tracker to `trackers.js`. Production
  keeps the "Coming soon" teaser; the route falls back to it when the flag is off.
  The backend API stays available regardless — only the UI is gated.

## Tests (#121, #126)
- Backend: transaction CRUD, year/month filtering, summary aggregation, per-type
  category + positive-amount validation, and cross-tenant IDOR.
- Frontend: `@testing-library/react` coverage for the api wiring, the
  transactions panel (incl. the type→category dependency), the dashboard, and the
  page (87 frontend tests total).
