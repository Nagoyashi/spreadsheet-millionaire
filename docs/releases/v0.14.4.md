# v0.14.4 — Typeable bounded inputs + docs status accuracy (2026-07-14)

User QA on the v0.14.3 production surface caught what the prod-wide sweep
missed: on Barista FIRE, the Assumptions number fields "jumped around" —
typing an age was outright impossible. One shared-component fix (#288) and a
docs accuracy pass (#289).

## The bug — min-clamp-per-keystroke made raised-min fields untypeable

`NumInput` (the shared input behind every calculator field) clamped both
bounds on every keystroke since the original input-hardening pass. That's
correct for `max` — typing appends digits, so a prefix of an in-range number
never overshoots it — but wrong for `min`: prefixes legitimately sit *below*
a raised min. With `min={18}`, typing "3" (on the way to "35") snapped to 18,
the next digit made "185", which overshot `max={70}` and snapped again.

The flaw shipped invisibly because the original four published calculators
only use `min={0}`; it surfaced once the age-bounded calculators went live.
A sweep of all 60 `NumInput` usages found the affected set:

- **Barista FIRE** — Current Age (18–70), the reported field
- **Coast FIRE** — Current Age (18–80), Retirement Age (30–80)
- **FIRE** — Withdrawal Rate (`min=0.1`: sub-1 entries like "0.5" were
  untypeable)
- Seven `min=1` year/month fields (Compound Interest, Inflation, Mortgage
  term, Emergency Fund months, Dividend, Investment Fee, Withdrawal Plan
  duration) — mild: a leading "0" snapped to 1

**Fix:** clamp `max` on change, `min` on blur. Negatives typed into a
non-negative range still snap immediately (they can never grow into range),
so `min={0}` monetary fields behave exactly as before. The robustness
invariant moves to the committed value: a blurred field is never outside
`[min, max]`. Verified unaffected: all tracker fields (`min: 0` throughout),
Sankey's own row inputs, TransactionsPanel's submit-time-clamped recurrence.

New `NumInput.test.jsx` pins the behavior (8 cases: digit-by-digit entry
through a raised min, blur-snap, max-on-change, `8e31` paste, negative paste,
empty-field semantics). Suite 184 → 192. DECISIONS.md § "Numeric input is
bounded and clamped" amended with the split-by-bound rationale.
