# v0.12.1 — Admin portal polish (2026-06-29)

Post-launch polish of the Admin Control Center (v0.12.0), from founder feedback.

## Trackers are now toggleable from the portal
The Net Worth and Income & Expense trackers appear in the Overview catalog and
publish/unpublish through the **same runtime mechanism as calculators** — so
revealing a tracker out of "coming soon" is an admin toggle, not a redeploy.

- New `backend/publishable.py`: `PUBLISHABLE_TYPES = VALID_CALC_TYPES + TRACKER_TYPES`
  (`net-worth`, `income-expenses`). `db_init` seeds a `calculator_publish` row per
  publishable type; trackers default **unpublished** (coming soon).
- The build-time tracker flags are **gone** — `frontend/src/featureFlags.js` is
  deleted. `trackers.js` now exposes `useLiveTrackers()` / `useVisibleUpcoming()`
  (runtime hooks over the published set), and the `/app/net-worth` +
  `/app/income-expenses` route guards check the published set instead of a flag.
- Overview lists the two trackers with a "Tracker" category pill and a
  "Coming soon" status until published.

## Superadmin role — only you can mint admins
- New `users.is_superadmin` flag sits above `is_admin`. A superadmin is implicitly
  an admin; only a superadmin can grant/revoke the admin role.
- `PATCH /api/admin/users/:id/admin { is_admin }` is `superadmin_required`
  (404-invisible to normal admins), audit-logged (`grant_admin`/`revoke_admin`),
  with guards against changing your own role or revoking a superadmin's admin.
- The Users screen shows a "Make/Revoke admin" action **only to a superadmin**,
  and a `SUPERADMIN` badge on superadmin accounts. Bootstrap is manual
  (`User.set_superadmin`) — like the first admin.

## Smaller items
- **Unpublishing keeps user data** (clarified + verified): the publish toggle only
  flips a flag; saved calculator data is never touched and reappears on republish.
- **Analytics revenue** — a `Revenue · MRR` KPI placeholder ("—", unlocks after
  billing), alongside Free→Paid.
- Tier control stays manual now and as a permanent override; when billing lands a
  subscription webhook will set `tier` automatically (documented, not yet built).

## Notes / migration
- Schema additions (`users.is_superadmin`, the tracker publish rows) are additive
  and apply automatically on deploy via migrate-on-boot.
- **Behaviour change (dev/staging):** trackers now default to "coming soon"
  everywhere (previously dev auto-revealed them via the build flag) — reveal them
  per-environment from `/admin`. Production was already "coming soon", so no
  end-user change there.
- To use `/admin` in an environment, bootstrap one superadmin once:
  `User.set_superadmin(<your_user_id>, True)` against that environment's DB.

See `DECISIONS.md` §§ "Runtime publish state" (trackers addendum) and "Superadmin
role".
