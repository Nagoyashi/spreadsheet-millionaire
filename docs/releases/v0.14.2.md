# v0.14.2 — CI confidence + scale sanity (2026-07-12)

The third and final Phase-14 sub-release — the gate that answers "is it solid
for go-live." No product changes at all: this release is tests, CI hardening,
and a load-check tool. The backend suite grew from 132 to 147 tests, every run
executed against real Postgres locally and in CI. **Phase 14 (go-live readiness
+ instrumentation) is complete with this release.**

## CI can no longer go green without the database (#183)
- Audit: the CI Postgres service container (from #27) and all six issue-#25
  gotchas (config-validation timing, SSL mode, Talisman, rate-limiter bleed,
  CSRF, per-request commit isolation) were already handled. The one real gap:
  if `TEST_DATABASE_URL` were ever lost, the DB-backed majority would silently
  skip and CI would still pass. Under `CI_REQUIRE_DB=1` (now set in CI) a
  missing URL is a hard failure; locally the skip convenience is unchanged.

## Migration-system tests (#184)
- The `db_init.py` contract pinned: re-running on a current schema is lossless;
  an **older** schema migrates forward (dropped later-release columns re-added
  with defaults backfilled); CHECK constraints rebuild from the Python source
  of truth; and publish-state seeding backfills missing rows but **never resets
  an admin's live toggle** — the one migrate-on-boot failure mode that would
  silently unpublish the site on deploy.

## Entitlement-boundary tests (#185)
- Every documented access-control claim pinned, including two that were
  documented but never tested: suspension is checked only **after** the
  password verifies (so a 403 can't probe which emails exist), and the
  admin/superadmin gates read fresh from the DB per request (suspending or
  revoking an admin takes effect on their very next request, live session or
  not). Also pinned deliberately: a suspended normal user's live session keeps
  app access (the documented login-gate design) and tiers carry no feature
  enforcement until billing.

## Deletion-cascade end-to-end test (#186)
- The #179 service tests proved the cascade contract; this adds the full-path
  proof: a real registered user with a row in every user-scoped table deletes
  their account through the actual `DELETE /api/auth/account` route (password +
  CSRF) — route guards, service, verified cascade, and session teardown in one
  test.

## Neon pooled-connection load check (#187)
- `backend/pooling_check.py` — a standalone script that drives N concurrent
  workers of the app's exact per-request pattern (connect → `SELECT 1` → close)
  against a `DATABASE_URL`, reporting latency percentiles, throughput, and
  errors, warning when a `neon.tech` host lacks the `-pooler` suffix (the
  classic direct-endpoint misconfig). Validated locally: 3,020 requests,
  604 req/s, zero errors. Running it against each environment's real pooled
  string is the documented operator step (`DEPLOYMENT.md` § 7).

## Auth rate limits pinned (#188)
- Audit: login (20/h; 5/min), register (10/h), forgot-password (3/h), and
  reset-password (10/h; 5/min) were already limited; "verify-resend" doesn't
  exist (no email-verification flow). The gap: nothing enforced that the limits
  stay — the main suite runs with limiting disabled, so a dropped decorator
  would never fail a test. A dedicated limits-live suite now drives each
  endpoint over its threshold (failure responses count too — attackers don't
  send valid requests).

## Verification
147 backend tests (15 new this release) on real Postgres, locally and in CI
under `CI_REQUIRE_DB=1`; frontend 165. No dependency changes.
