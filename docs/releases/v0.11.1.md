# v0.11.1 — Tracker polish & recurring transactions (2026-06-23)

A build-in-public follow-up to the v0.11.0 trackers: one shared app shell, real
enhancements to both tracker Overviews, and recurring transactions for the Income
& Expense tracker. The trackers still **ship dark** behind their flags — this is
all dev/staging-visible polish ahead of a later waiting-list launch.

## Shared collapsible sidebar (#137)
- One `AppSidebar` + `AppShell` now back the calculator landing, every calculator
  page, **and both trackers** (which previously had no sidebar). Replaces the two
  earlier forks.
- Three **sibling** top-level categories, same tier: Calculators (expandable to
  its calculators, shown muted), Net Worth, and Income & Expenses — each
  flag-gated, so trackers still appear only when revealed.
- A collapse/expand toggle (full ↔ icon rail) whose state persists across
  navigation via an in-memory hook (no storage, no global store).

## Income & Expense — Overview + recurring transactions (#137, #141)
- **Overview:** per-month income with an average ⇄ median toggle (median ignores
  empty months so a partial year isn't dragged down), and a spending-by-category
  pie that re-scopes to a single month (month picker or click a bar).
- **Recurring transactions:** a transaction can repeat — daily / weekly / monthly
  / yearly, plus custom intervals (e.g. "every 2 weeks"). Stored as a
  `(recurrence_unit, recurrence_interval)` pair on the row.
- **Cashflow forecast:** recurring rows are projected into the empty future months
  of the chart as visually-distinct (lighter) forecast bars. Projections are
  derived at read time and never persisted.

## Net Worth — Overview polish (#140)
Read-side only — every figure derives from existing data, no schema change.
- **Net Worth Over Time:** the change vs the previous snapshot ("+$X since last
  snapshot", sign-coloured), plus a hint that a second snapshot unlocks the trend.
- **Debt-to-asset ratio** under Total Liabilities.
- **Liabilities breakdown** — by type plus a "Mortgages" slice from real-estate
  balances, so it reconciles with total liabilities.
- **Per-item gain/loss** column in each asset class tab; the formulas mirror the
  backend exactly, so per-item figures sum to the aggregate return.

## Compact footer everywhere (#137)
- The in-app footer is now a single row (© line + Privacy / Terms / Imprint /
  Source) rendered once by `AppShell`, so it appears on the calculator grid, every
  calculator page, and both trackers — without dominating the page.

## Storage & migrations
- `ie_transactions` gains `recurrence_unit` + `recurrence_interval` via an
  **idempotent DDL migration** in `db_init.py` (`ALTER ADD COLUMN IF NOT EXISTS` +
  CHECK rebuild); existing rows default to a one-off. This is the normalised-table
  path — **not** a client saved-data version bump (Hard Rule #5 is scoped to opaque
  blobs). **Deploys must run `cd backend && python db_init.py`** against the target
  database before the new code serves writes.

## Tests
- Backend: recurrence column round-trip, one-off normalisation, and validation
  (against Postgres in CI).
- Frontend: forecast projection, the Income & Expense Overview selectors, and the
  Net Worth Overview selectors (gain reconciliation, debt ratio, snapshot delta,
  liabilities breakdown). 106 frontend tests total.
