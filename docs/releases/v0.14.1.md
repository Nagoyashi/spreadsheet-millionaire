# v0.14.1 — Data lifecycle + legal (2026-07-12)

The second of three Phase-14 sub-releases (go-live readiness). The theme is the
full lifecycle of a user's data — provable erasure, portable export, truthful
legal pages, and a support path for requests that arrive by email. One design
thread ties it together: a single authoritative registry of user-scoped tables
(`services/account_deletion.USER_SCOPED_TABLES`) now drives deletion, export,
the admin tools, and the tests — there is no second list anywhere, and schema
drift fails loudly instead of silently orphaning financial rows.

## Account deletion — explicit, verified cascade (#179)
- The schema's `ON DELETE CASCADE` already wiped user rows, but implicitly:
  nothing owned the table enumeration and nothing verified it. Deletion now goes
  through a single service that counts rows per table (the erasure report),
  deletes the user row, and **verifies every user-scoped table is empty before
  committing** — any surviving row rolls the whole deletion back rather than
  committing a half-wipe.
- A schema-drift guard (`verify_cascade_coverage()`) cross-checks the live
  database against the registry in both directions; the deliberate `SET NULL`
  retainers (`admin_audit_log`, `calculator_publish.updated_by`) were verified
  PII-free, so their retention is GDPR-clean.
- `User.delete` is gone — the service is the only deletion path.

## Data export — "download my data" (#180)
- `GET /api/auth/account/export` returns everything stored for the authenticated
  user — account details, saved calculations, all tracker entries — as one
  versioned JSON attachment (GDPR Art. 15/20, the read counterpart to Art. 17
  deletion). Self-service from the new Settings "Your data" section.
- The table list derives from the deletion registry; secret columns
  (`password_hash`, `token_hash`) are stripped while the rows that hold them
  remain. GET + login only, rate-limited — no password gate, since a live
  session can already read all of this through the normal APIs.

## Legal pages current with the product (#181)
- Privacy Policy: tracker entries added to "what we collect"; access/portability
  now names the self-service export; deletion describes the verified full-table
  erasure; retention covers tracker data and discloses the PII-free audit-record
  retention.
- Terms: the load-bearing "not financial advice" clause now covers calculators
  **and trackers**; "your content" covers tracker entries and the export.
- Deletion copy made truthful everywhere: the confirm modal and the Settings
  danger zone say all data (calculations and tracker entries) is deleted, and
  point at the export for a copy-first.

## Admin support tools (#182)
- `GET /api/admin/users/:id/export` and `DELETE /api/admin/users/:id` — the
  support path for GDPR requests that arrive by email. Both reuse the #179/#180
  services verbatim; there is no second deletion or export code path.
- Both are **audit-logged**, including the export — an admin reading a user's
  data is precisely the event the audit log exists for (details carry per-table
  row counts, never the data).
- Delete guards mirror the existing privilege model: not yourself, never a
  superadmin, an admin account only by a superadmin. The Users screen gains an
  "Export data (JSON)" action and a two-step, non-optimistic delete confirm.

## Verification
Backend suite grown to 132 tests (13 new across deletion / export / admin
tools), run against real Postgres locally and in CI; frontend 165.
