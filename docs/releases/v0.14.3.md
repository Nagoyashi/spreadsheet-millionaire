# v0.14.3 — Calculator math fixes from the prod-wide sweep (2026-07-14)

A manual QA sweep of the full production surface (pre-billing readiness) —
all 12 calculators anonymously, then the trackers and account flows on a
test account — found four real calculator math bugs, one tracker-blocking
form bug, and a set of input-hardening gaps — all fixed here (#279, #282).
Every math fix is pinned by new unit tests with closed-form expectations;
the frontend suite grew from 165 to 184.

## Math fixes

- **Compound Interest — compounding frequency was inverted.** The per-period
  contribution was `pmt·(n/12)` instead of `pmt·(12/n)`, so annual
  compounding invested $41.67/yr instead of $6,000 (showing *negative*
  interest next to a correct Total Contributed), and daily invested
  $15,208/day (a $242M "final value"). Monthly — the default — was the only
  correct case, which is why it survived unnoticed.
- **Withdrawal Plan — three bugs in one stat row.** The Real Return card
  rendered the raw fraction (5% return − 2.5% inflation displayed "0.0%");
  "Balance at Nyrs" was read after the full 50-year simulation instead of at
  year N ($12M instead of $4.3M with zero withdrawals); and any depletion
  inside the 50-year sim — even past the user's horizon — forced
  "$0 / Depleted" to render beside "Portfolio Lasts 41 yrs". The card now
  reports the true year-N balance, and the status box distinguishes
  "lasts your horizon (runs out later, in year X)" from "unsustainable".
- **Coast FIRE — coast age compared against today's threshold.** The
  years-to-coast loop checked the growing balance against the *static*
  current coast number, but the threshold rises every year (fewer coasting
  years remain). Zero-contribution scenarios "coasted" in 10 years — 
  mathematically impossible, since balance and threshold grow at the same
  rate. The loop now crosses the per-year threshold, never-on-track renders
  a "Not on track" state, a retirement age in the past clamps to "need the
  full FIRE number now", and the chart's coast line shows the rising
  threshold.
- **FIRE — unreachable goals showed "0 yrs / Retiring 2026".** With zero
  annual savings the simulation was skipped entirely and `years` stayed 0.
  The simulation now always runs (an existing portfolio can compound to the
  target with no contributions) and returns the "100+" state when a century
  isn't enough.

## Input hardening & polish

- **Mortgage:** a down payment above the home price clamps the loan to $0
  instead of producing a negative monthly payment.
- **Sankey:** typed/pasted negative amounts clamp to 0 (native `min` only
  guards the spinner); item/group inputs gained `aria-label`s.
- **Inflation:** clearing every field no longer renders "NaN%".
- **NumInput (all 12 calculators):** the visible label is now
  programmatically associated with its input (`useId` + `htmlFor`) — screen
  readers previously announced nothing but the "$"/"%" adornment.
- Empty-state label fragments ("in yrs", "At % fee") fall back to 0.

## Net Worth tracker — category form permanently disabled after tab switch (#282)

`WealthPage` reuses one `CategoryManager` instance across the five category
tabs, but the add/edit form state was seeded only on mount — after switching
tabs the previous tab's field shape lingered in state, `canSubmit()` could
never pass, and the **Add button never enabled**. Users could only add items
to whichever category tab they opened first per page load. The form now
re-seeds whenever the category changes. Bonus: "Add liabilitie" → "Add
liability" via a config-level singular label.

## Verification

184 frontend tests (19 new regression tests, closed-form expectations for
the math), lint and build clean. Compound and FIRE additionally verified
end-to-end in the browser against the PR preview; the tracker fix verified
against the live prod repro (React state inspection + reload-first-tab
control).
